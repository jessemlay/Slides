What is git?

Git is a distributed revision control system.

Git features

  • Every developer has there own copy
  • Changes can be stored locally until your ready to commit
  • A repo can be tagged, creating a snapshot of the current state of the repo
  • Branching is lightweight and the recommended way to work on changes

Workflow

Git is very flexible and can be used in a variety of ways, but lets look at a simple workflow for small teams

Two developers are task with working on a new feature in the same repository, for this example we will call them John and Marry

Cloning

You can thinking of cloning as a SVN Checkout, except you grab a independent repo you can work on

git clone http://jlay@localhost:8080/gitserver/SMARTTagDocs.git

When you clone a repo, git will automatically create a link to the origin so you can push upstream

Local Repositories

John and Mary both clone the repo and can now work independently on their respective features

            
git status # View the state of the repo
git add  # Stage a file
git commit # Commit a file
Note: this will only commit to the local repo and will not affect the main repository

Pushing

Pushing is like a SVN Check-in. John has finish his feature and is ready to commit to the main branch

git push origin master

This will push his changes to the main repository

Mary has finish her feature and pushes to master

git push origin master

Since her local history has diverged from the central repository Git will refuse the commit. Mary must rebase on top of Johns commits. The important part of this command is the --rebase which tell Git to to move all of Mary's commites to the tip of the master brance keeping a linear commit log

git pull --rebase origin master

Feature Branches

The previous workflow was a very basic example

Feature branches

Creating a branch per feature allows developer to work independently of the master or develop branch

Once the feature is ready a pull request can be made allowing the reviewer to accept the request or provide comment and reject it

How It Works

The Feature Branch Workflow still uses a central repository, and master still represents the official project history. But, instead of committing directly on their local master branch, developers create a new branch every time they start work on a new feature. Feature branches should have descriptive names, like animated-menu-items or issue-#1061. The idea is to give a clear, highly-focused purpose to each branch.

Git Submodules

Our current repository is one single repo, which is nice cause when you update it will grab everything. However it can also show changes not part of the code you are working on and do not want to commit. Git modules allow you to link one repo to another. For instance you could set SMART_Library as a submodule to Admin Portal
